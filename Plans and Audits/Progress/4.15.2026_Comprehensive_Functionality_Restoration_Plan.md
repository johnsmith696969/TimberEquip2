# Comprehensive Functionality Restoration Plan

Date: 2026-04-15
Scope: Restore and improve public browsing, listing detail UX, admin/super-admin access, taxonomy management, auctions, PostgreSQL/Data Connect readiness, Cloud SQL operations, accessibility, security, SOC 2 observations, and retention controls without reducing current functionality.

## Implementation Progress

- 2026-04-15: Started P0 implementation slice.
- Completed equipment-hours icon cleanup for listing cards, listing detail, profile inventory, and compare specs; clocks remain only where they represent actual time/member-since/governance timing.
- Added centralized search page copy generation for category, subcategory, manufacturer, and query-intent headings.
- Updated subcategory routes to retain parent category context so pages can say `Logging Equipment Forwarders For Sale` and apply both parent category and subcategory filters.
- Changed visible inventory count copy to `{count} Machine(s) Available`.
- Made Fuse.js lazy so filter-only browsing does not build the fuzzy search index.
- Hardened runtime taxonomy removal/add flows with case-insensitive canonical matching, deep-cloned removal records, and dealer/listing auto-adds that cannot create new top-level categories.
- Verification: `npm run test -- searchPageCopy.test.ts equipmentTaxonomy.test.ts` passed; `npm run build` passed. `npm run lint` did not complete within 6 minutes in this local workspace.

## Guiding Rules

- Ship in small, reversible patches. Do not combine auth, database, auction, and visual changes in one deploy.
- Keep Firestore as the production source of truth until PostgreSQL parity is verified with consistency checks.
- Treat PostgreSQL/Data Connect as shadow/read-model infrastructure first, then move selected reads after validation.
- Preserve user-facing routes, SEO pages, dealer storefronts, saved searches, email unsubscribe, and listing creation flows during every change.
- Add tests or a Playwright smoke path for each bugfix before marking it complete.
- For data retention, implement 90-day minimum recoverability for deleted accounts and operational records, but redact/minimize personal data where privacy deletion rights require it. Longer retention is allowed only where there is a documented legal, fraud, tax, billing, audit, or safety basis.

### TASK-AUTH-001 - Fix `caleb@forestryequipmentsales.com` super-admin authorization drift

Observed likely causes:
- Server trusts custom claims first, then `PRIVILEGED_ADMIN_EMAILS`, then Firestore profile role.
- Client strips role from local profile cache by design, so UI can look refreshed while server token is stale.
- A role changed in Firestore does not automatically update Firebase custom claims.
- ID tokens need refresh after claim changes.
- `PRIVILEGED_ADMIN_EMAILS` must exist in the deployed Functions/Cloud Run environment or Secret Manager, not just local docs.

Code targets:
- `TimberEquip-main/functions/index.js`
- `TimberEquip-main/src/components/AuthContext.tsx`
- `TimberEquip-main/src/services/adminUserService.ts`

Implementation:

  - token uid/email
  - token custom claims role
  - Firestore profile role/status
  - privileged-email match
  - allowed dashboard scopes
  - token issued/auth time
- Add an admin bootstrap repair action:
  - set Firebase custom claims for privileged admin emails
  - ensure Firestore profile role/status/access source match
  - force client token refresh after repair
- Surface exact authorization reason in admin UI instead of generic unauthorized.

Acceptance criteria:
- `caleb@forestryequipmentsales.com` can access Users, Content, Billing, Taxonomy, and Auctions tabs.
- Diagnostics show token claims and Firestore profile are aligned.
- Role changes prompt a token refresh or re-login.

### TASK-TAX-001 - Fix taxonomy removal persistence

Problem:
Removing a taxonomy model/manufacturer does not reliably remove it.

Likely causes:
- Removal checks use exact object paths while runtime taxonomy is merged from static JSON, static `equipmentData.ts`, and overrides.
- Case/canonical mismatch can make removal appear successful but still reappear from bundled taxonomy.
- Nested `removals` cloning is shallow for nested maps, risking mutation bleed.

Code targets:
- `TimberEquip-main/src/services/taxonomyService.ts`
- `TimberEquip-main/src/components/admin/TaxonomyManager.tsx`

- When removing a manufacturer, remove all child model overrides under that manufacturer.
- Add tests for removing bundled manufacturer/model, runtime manufacturer/model, and mixed-case labels.

Acceptance criteria:
- Removed bundled manufacturer stays removed after reload.
- Removed runtime manufacturer stays removed after reload.
- Removed model stays removed after reload and does not reappear from JSON fallback.
- Taxonomy manager refreshes dependent dropdowns immediately.

## P1 - Data Retention, Deletion, and SOC 2 Section 8

### TASK-RET-001 - Replace destructive account deletion with compliant retention workflow

Problem:
Current user delete route hard-deletes listings, inquiries, invoices, subscriptions, consent logs, the user profile, and Firebase Auth user. This conflicts with requested 90-day recoverability and may destroy audit/billing/legal evidence.

Code targets:
- `TimberEquip-main/src/server/routes/user.ts`
- `TimberEquip-main/functions/index.js`
- `TimberEquip-main/src/pages/Profile.tsx`
- Firestore rules
- PostgreSQL marketplace user delete mutations

Implementation:
- Account deletion request immediately:
  - disables login or marks `accountStatus=deletion_pending`
  - removes public profile/storefront visibility
  - hides active listings
  - revokes sessions where possible
  - writes `deletedAt`, `deleteRequestedAt`, `retentionHoldUntil = deletedAt + 90 days`
- Do not hard-delete operational records immediately.
- Redact direct profile PII where no exception applies.
- Retain audit, billing, payment, security, fraud, and legal records according to retention policy.
- Add restore path for accidental deletion within 90 days.
- Add hard-delete/purge job after retention hold expires and legal holds are clear.

Acceptance criteria:
- Deleted accounts are recoverable for at least 90 days.
- Public site no longer exposes deleted account profiles/listings.
- Audit trail and billing history remain intact.
- Privacy deletion requests receive clear status and basis for retained records.

Legal note:
This needs counsel review. The California Attorney General says consumers may request deletion, subject to exceptions, and Minnesota residents have the right to request deletion under the MCDPA. Retention should be documented by purpose and minimized to what is necessary.

### TASK-RET-002 - Data retention policy matrix

Implementation:
- Create machine-readable retention config:
  - users/profile PII: 90-day recoverable hold, then redact/purge unless legal hold.
  - account audit logs: 2+ years minimum.
  - billing/invoices/tax/payment metadata: 7 years unless counsel says otherwise.
  - consent logs: 2+ years.
  - security logs: 1 year online, 2+ years archive.
  - deleted listings/media: 90-day restore hold (add restore options to roles, only allow if listings are available for use verified by subscription), then delete or anonymize if not tied to transaction/dispute.
  - auction bids/invoices: 7 years or legally required period.
- Update Privacy Policy and Terms to match actual retention behavior.

Acceptance criteria:
- Retention matrix exists in repo.
- Purge job reads policy from config.
- Admin can put account/listing/invoice on legal hold.
